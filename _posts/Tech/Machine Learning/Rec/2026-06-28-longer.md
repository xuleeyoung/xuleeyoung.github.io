# LONGER: Scaling Up Long Sequence Modeling in Industrial Recommenders

## 一、背景与动机

### 1.1 超长用户行为序列的价值

在现代工业推荐系统中，用户行为序列的长度已成为决定模型效果的关键因素之一。传统推荐模型通常依赖较短的序列（$10^2$ 量级），但随着用户数据的积累和技术基础设施的升级，研究者和工程师们开始探索更长序列（$10^3$ 甚至 $10^4$ 以上）的建模价值。

超长用户行为序列能够同时捕捉用户的**长期偏好**和**短期意图**：

- **长期偏好**：用户稳定的兴趣特征，如消费能力、品牌偏好、生活习惯等
- **短期意图**：用户当前的即时需求，如节假日促销、突发热点、季节性需求等

通过融合超长序列，模型能够更精准地理解用户全生命周期的行为模式，从而提供更加个性化和准确的推荐服务。

### 1.2 现有方案的局限性

尽管超长序列建模具有重要价值，但现有方案在实际应用中仍面临诸多挑战：

| 方案类型 | 代表工作 | 主要缺陷 |
|---------|---------|---------|
| 两阶段检索 | SIM / TWIN | 从超长序列筛选Top-k再做短序列建模，上下游不一致性严重 |
| 预训练用户嵌入 | Pre-trained User Embeddings | 仅支持 $\leq 10^3$ 的预训练长度，无法充分利用超长序列 |
| 记忆增强模型 | MIMN / LMN / MARM | 需要长期训练积累命中率，冷启动效果差 |

**两阶段检索的固有问题**：第一阶段的检索模型和第二阶段的序列建模模型优化目标不一致，导致语义漂移。例如，检索模型可能选择与目标物品语义相似的高曝光物品，但这些物品并不一定能代表用户的真实偏好。

**预训练方案的瓶颈**：受限于计算资源和注意力机制的二次复杂度，预训练阶段通常只能处理较短的序列，限制了模型捕捉超长期偏好的能力。

**记忆增强的冷启动问题**：基于记忆网络的方案需要在线上服务过程中持续积累用户行为的"命中率"，对于新用户或长尾用户效果不佳。

### 1.3 研究灵感与硬件支撑

LONGER的研究团队从两个维度获得了启发：

1. **LLM缩放定律的启示**：大语言模型证明了"更大规模、更长上下文"能够带来显著的性能提升，这一规律在推荐系统中同样适用。

2. **GPU基础设施的升级**：现代GPU（如NVIDIA A100/H100）提供了更大的显存容量和更高的计算吞吐，使得在工业场景中处理超长序列成为可能。

基于以上分析，LONGER团队提出了一种端到端的超长序列建模框架，旨在突破现有方案的局限性，在工业规模下实现高效的超长用户行为建模。

## 二、核心架构详解

LONGER的架构设计围绕三个核心模块展开：**全局令牌机制**、**令牌合并模块**和**混合注意力机制**。这三者协同工作，共同解决了超长序列建模中的核心挑战。

### 2.1 全局令牌机制（Global Tokens）

#### 2.1.1 设计动机

在处理超长序列时，标准自注意力机制面临一个严峻问题：**信息聚合效率低下**。随着序列长度增加，模型需要从数万甚至数十万个物品中提取与目标物品相关的信息，这在计算上是低效的。

借鉴LLM领域的研究成果（如StreamLLM），LONGER引入了**全局令牌（Global Tokens）** 作为信息聚合的锚点。

#### 2.1.2 全局令牌的组成

全局令牌由以下四部分组成：

```python
# 全局令牌的组成示意
GlobalTokens = {
    "target_item": target_item_embedding,      # 目标物品表示
    "cls_token": learnable_cls_embedding,     # 可学习的CLS令牌
    "uid_embedding": user_id_embedding,        # 用户ID嵌入
    "high_order_features": compress_interactions # 高阶压缩用户-物品交互特征
}
```

1. **目标物品表示（Target Item Representation）**
   - 编码当前候选物品的语义信息
   - 作为注意力计算的参照点

2. **可学习CLS令牌（Learnable CLS Token）**
   - 类似BERT的[CLS]令牌
   - 通过预训练学习最优的信息聚合方式
   - 提供一个与序列无关的全局视角

3. **用户ID嵌入（UID Embedding）**
   - 编码用户的身份特征
   - 捕捉用户的长期稳定偏好
   - 缓解冷启动用户的序列稀疏问题

4. **高阶压缩用户-物品交互特征**
   - 对原始序列进行初步压缩
   - 提取高阶的交互统计特征
   - 如：用户的点击序列中各类别的TF-IDF、用户的活跃度变化趋势等

#### 2.1.3 稳定性作用

全局令牌的核心作用是**稳定长序列的注意力分布**：

- **缓解注意力sink效应**：在标准自注意力中，模型可能过度关注序列中的某些特定位置（如序列开头），导致信息利用不均。全局令牌提供了一个额外的"注意力锚点"，使得注意力分布更加均匀。
- **作为信息聚合的瓶颈**：所有与目标物品相关的信息都需要通过全局令牌进行汇聚，形成一个紧凑的序列表示。

### 2.2 令牌合并模块（Token Merge）

#### 2.2.1 问题定义

标准自注意力的计算复杂度为 $O(L^2 d)$，其中 $L$ 是序列长度，$d$ 是隐藏维度。当 $L$ 达到数千甚至数万时，计算代价变得不可承受。

**复杂度分析示例**：

```
场景：L = 2048, d = 32
标准注意力 FLOPs = 2 × L² × d = 2 × 2048² × 32 ≈ 268M
```

#### 2.2.2 合并策略

LONGER采用**令牌合并（Token Merge）**策略，按压缩因子 $K$ 对序列长度进行压缩：

```
原始序列长度: L
压缩后序列长度: L' = L / K
```

#### 2.2.3 合并方式

LONGER支持两种令牌合并方式：

**方式一：简单拼接（Concatenation Merge）**

```
[token_i, token_{i+K}, token_{i+2K}, ...] → merged_token_i
```

- 将间隔为 $K$ 的 $K$ 个令牌拼接为一个更长的向量
- 计算量：$O((L/K)^2 \times Kd)$ = $O(L^2d/K)$
- 优点：实现简单，几乎无额外计算开销
- 缺点：丢失了组内令牌之间的交互信息

**方式二：组内交互（InnerTransformer Merge）**

```
[token_i, token_{i+K}, token_{i+2K}, ...] → InnerTransformer → merged_token_i
```

- 在合并前，先用一个轻量级的InnerTransformer捕捉组内交互
- 计算量：$O(L \times K^2 \times d_{inner})$
- 优点：保留了组内的细粒度交互信息
- 缺点：额外引入一部分计算开销

#### 2.2.4 复杂度优化效果

以 $L=2048, d=32, K=4$ 为例，对比如下：

| 方案 | 序列长度 | FLOPs | 节省比例 |
|------|---------|-------|---------|
| 标准注意力 | 2048 | 587M | - |
| 简单拼接合并 | 512 | 151M | 74.3% |
| InnerTransformer合并 | 512 | 336M | 42.8% |

可以看到，令牌合并策略显著降低了计算复杂度，使得超长序列的处理成为可能。

### 2.3 混合注意力机制

LONGER的注意力机制采用**全局令牌 + 采样序列令牌**的混合模式，并结合了**交叉注意力**和**自注意力**的级联结构。

#### 2.3.1 输入构建

```
Input = [GlobalTokens] + [Sampled Sequence Tokens]
       = [全局令牌] + [采样序列令牌]
```

**采样策略**：从超长序列中按重要性或均匀间隔采样部分令牌，与全局令牌拼接作为注意力计算的输入。

#### 2.3.2 位置编码设计

LONGER采用**双位置编码**机制：

1. **绝对时间差编码（Absolute Time Difference）**
   - 计算每个序列令牌与目标物品的绝对时间差
   - 编码形式：$\text{PE}(t) = \text{positional\_encoding}(\Delta t)$
   - 作用：捕捉用户行为的时间衰减特性

2. **可学习绝对位置嵌入（Learnable Absolute Position）**
   - 与标准Transformer中的位置编码类似
   - 直接学习每个位置的最优表示
   - 作用：提供序列顺序的先验知识

#### 2.3.3 注意力层级结构

LONGER采用**两阶段注意力**结构：

```
Layer 1: Cross-Causal Attention (交叉因果注意力)
Layer 2~N: Self-Causal Attention (自因果注意力)
```

**第一层：交叉因果注意力**

```python
# 伪代码：交叉因果注意力
def cross_causal_attention(query, key, value, mask):
    """
    序列令牌可以看到全局令牌，但序列令牌之间互不可见
    这是为了支持后续的KV缓存服务
    """
    # Q = 序列令牌, K/V = [全局令牌 + 序列令牌的历史K/V]
    attention_weights = softmax(Q @ K^T + mask)
    output = attention_weights @ V
    return output
```

**关键特性**：因果掩码确保序列令牌无法"看到"候选物品，这为后续的KV缓存服务奠定了基础。

**后续层：自因果注意力**

```python
# 伪代码：堆叠的自因果注意力
def self_causal_attention(hidden_states, num_layers):
    """
    捕捉序列内部的依赖关系
    """
    for layer in range(num_layers):
        hidden_states = self_attention_layer(hidden_states)
    return hidden_states
```

**设计理由**：首层使用交叉注意力，可以让序列令牌直接与全局令牌交互；后续层使用自注意力，逐步捕捉序列内部的复杂依赖关系。

## 三、训练与部署优化

### 3.1 全同步训练框架

工业级推荐系统通常面临**模型规模大、训练数据海量**的挑战。LONGER采用**全同步训练（Full Synchronous Training）**框架，实现软硬件协同设计。

#### 3.1.1 分层存储架构

LONGER采用三层存储架构，根据数据访问频率分配存储介质：

```
┌─────────────────────────────────────────────────────────────┐
│                    GPU HBM (高频访问)                         │
│  - 当前训练批次的数据                                          │
│  - 梯度、优化器状态                                            │
│  - 容量：80GB (A100)                                          │
├─────────────────────────────────────────────────────────────┤
│                    CPU MEM (中频访问)                         │
│  - 历史批次的数据缓存                                          │
│  - 预取下一批次的数据                                          │
│  - 容量：TB级                                                  │
├─────────────────────────────────────────────────────────────┤
│                    SSD (低频访问)                            │
│  - 归档的历史数据                                              │
│  - 不常用的检查点                                              │
│  - 容量：PB级                                                  │
└─────────────────────────────────────────────────────────────┘
```

#### 3.1.2 数据流水线

```python
# 伪代码：三层存储数据加载
def data_loader(sequence_id):
    # Step 1: 检查GPU HBM缓存
    if sequence_id in gpu_cache:
        return gpu_cache[sequence_id]
    
    # Step 2: 检查CPU MEM缓存
    if sequence_id in cpu_cache:
        data = cpu_cache[sequence_id]
        prefetch_to_gpu(data)
        return data
    
    # Step 3: 从SSD加载
    data = ssd_loader.load(sequence_id)
    cache_to_cpu(data)
    prefetch_to_gpu(data)
    return data
```

### 3.2 混合精度训练与激活重计算

#### 3.2.1 混合精度训练

LONGER采用**BF16/FP16混合精度**训练策略：

| 精度类型 | 使用场景 | 优势 |
|---------|---------|------|
| BF16 | 前向传播、梯度计算 | 更大的动态范围，避免溢出 |
| FP16 | 优化器状态 | 节省50%显存 |
| FP32 | 关键操作（如Softmax、LayerNorm） | 保持数值稳定性 |

**性能提升**：混合精度训练平均提升训练吞吐量约18%。

#### 3.2.2 激活重计算

激活重计算（Activation Recomputation）是减少显存占用的经典技术：

```
标准流程：前向传播 → 保存所有激活 → 反向传播 → 释放显存
激活重计算：前向传播 → 释放激活 → 反向传播时重新计算 → 无需保存所有激活
```

**权衡**：牺牲少量计算（约20%）换取显著显存节省（约50%）。

### 3.3 KV缓存服务

KV缓存是LONGER实现高效在线服务的关键技术创新。

#### 3.3.1 核心思想

在离线阶段预计算并缓存用户序列的Key和Value矩阵，在线服务时直接复用：

```
传统方案（每个候选物品需要）：
  Q = candidate @ W_q
  K = sequence @ W_k  (重新计算！)
  V = sequence @ W_v  (重新计算！)
  attention = softmax(Q @ K^T / √d) @ V

LONGER方案（用户序列的K/V只计算一次）：
  K_cached = sequence @ W_k  (离线预计算)
  V_cached = sequence @ W_v  (离线预计算)
  在线时：
  Q = candidate @ W_q
  attention = softmax(Q @ K_cached^T / √d) @ V_cached
```

#### 3.3.2 服务效率提升

| 指标 | 传统方案 | LONGER KV缓存 | 提升 |
|------|---------|---------------|------|
| 在线计算量 | 高（重复计算K/V） | 低（复用缓存） | ~85% |
| 吞吐量下降 | 40% | 6.8% | **6倍** |

KV缓存的引入使得LONGER能够在工业环境中实现高效的在线服务，将吞吐量损失从40%降低到6.8%。

## 五、与相关工作的对比

### 5.1 与RankMixer的关系

LONGER和RankMixer是字节跳动推荐系统研究的两项重要工作，它们在系统中扮演不同的角色：

| 模块 | 职责 | 关注点 |
|------|------|--------|
| **LONGER** | 序列建模 | 用户行为序列的表示学习 |
| **RankMixer** | 特征交互 | 多模态特征的交叉建模 |

**互补性**：
- LONGER输出的序列表示可以作为RankMixer的输入
- 两者独立扩展，可以根据业务需求独立优化
- 在字节跳动的生产系统中，LONGER和RankMixer通常联合使用

### 5.2 与统一架构的对比

部分后续工作（如OneTrans、HyFormer）尝试将序列建模和特征交互统一到一个框架中：

- **优势**：架构更简洁，可能存在协同优化空间
- **劣势**：增加了模型复杂度，可能难以同时在两个任务上达到最优

LONGER团队认为，解耦设计更具灵活性，能够针对不同任务进行独立优化，在工业场景中更易维护和迭代。

## 六、技术洞见

### 6.1 关键设计原则

从LONGER的设计中，我们可以提炼出以下关键原则：

1. **端到端优于多阶段**
   - 两阶段检索方案虽然简化了单阶段计算，但引入了语义漂移问题
   - 端到端建模能够保证上下游优化目标一致

2. **计算换内存的策略**
   - 激活重计算、KV缓存等技术体现了"以计算换内存"的思路
   - 随着GPU计算能力的提升，这一策略越来越有价值

3. **全局锚点的重要性**
   - 全局令牌作为信息聚合的锚点，是处理超长序列的关键
   - 类似的思路在LLM领域也被广泛采用（如StreamLLM）

### 6.2 工业落地的关键因素

LONGER能够成功落地字节跳动数十亿用户场景，关键因素包括：

1. **软硬件协同设计**：充分利用GPU HBM/CPU MEM/SSD的分层架构
2. **工程优化与算法创新并重**：混合精度、KV缓存等工程优化与算法设计同等重要
3. **渐进式迁移**：从单一场景逐步扩展到全量，避免大规模风险

### 6.3 未来展望

LONGER的工作为超长序列建模在工业推荐系统中的应用开辟了新的方向：

- **更长的序列**：探索 $10^5$ 甚至 $10^6$ 级别的序列建模
- **多模态融合**：将文本、图像、视频等多模态信息融入超长序列
- **跨平台迁移**：将LONGER的设计经验迁移到其他领域（如搜索、广告、对话系统）

## 七、总结

LONGER是字节跳动在工业推荐系统超长序列建模领域的重要突破。通过全局令牌机制、令牌合并模块和混合注意力机制的协同设计，LONGER成功实现了在数十亿用户规模下的端到端超长序列建模。

**核心技术贡献**：

1. **全局令牌机制**：稳定注意力分布，缓解长序列建模的信息聚合问题
2. **令牌合并模块**：将注意力复杂度从 $O(L^2d)$ 降低到 $O(L^2d/K)$
3. **KV缓存服务**：将在线服务吞吐量损失从40%降低到6.8%
4. **全同步训练框架**：支持工业级大规模稀疏模型的高效训练

LONGER的成功不仅在于算法创新，更在于将算法创新与工程优化深度融合，为工业级AI系统的设计与部署提供了宝贵的实践经验。

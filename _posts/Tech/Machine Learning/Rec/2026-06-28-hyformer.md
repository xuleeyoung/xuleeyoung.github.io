# HyFormer: 统一长序列建模与特征交互的混合型Transformer架构

## 摘要

HyFormer 是字节跳动 AML 与 Search 团队于 2026 年 1 月发布的 CTR 预测模型，作为 RankMixer 和 LONGER 的统一后续工作，提出了将长序列建模与异构特征交互深度融合于单一 Transformer 框架的创新范式。本文将详细解析其核心设计与技术洞见。

---

## 1. 背景与动机

### 1.1 工业级大规模推荐模型的核心挑战

现代工业级推荐系统需要在**严格效率约束**下同时处理两类核心挑战：

| 挑战类型 | 具体表现 |
|---------|---------|
| **超长用户行为序列** | 数十甚至数百级的用户历史行为（点击、购买、浏览等） |
| **异构非序列特征** | 用户画像、物品属性、上下文特征等多模态信息 |
| **严格效率约束** | 推理延迟要求（通常 < 50ms）、计算资源限制 |

### 1.2 解耦式范式的局限性

当前主流工业方案采用**"先压缩后融合"**的解耦式流程：

```
LONGER (序列压缩) → RankMixer (特征融合)
```

这种范式存在固有局限：

1. **序列压缩损失语义**：压缩操作不可避免地丢弃细粒度行为信息
2. **特征交互受限**：分离的两个阶段阻碍了序列与稠密特征的深层交互
3. **优化目标分散**：两阶段独立优化，难以端到端协同

### 1.3 统一建模的诉求

HyFormer 重新审视这一问题：能否在**单一统一框架**内同时完成序列建模与特征交互？

## 2. 核心架构解析

### 2.1 整体架构：统一混合Transformer

HyFormer 的核心设计理念是**摒弃解耦范式**，将长序列建模与特征交互深度集成于单一 Transformer 架构：

```
┌─────────────────────────────────────────────────────┐
│                    HyFormer                          │
│  ┌─────────────────────────────────────────────┐    │
│  │           Hybrid Transformer Layer           │    │
│  │  ┌─────────────────┐  ┌─────────────────┐    │    │
│  │  │ Query Decoding  │↔│ Query Boosting  │    │    │
│  │  └─────────────────┘  └─────────────────┘    │    │
│  └─────────────────────────────────────────────┘    │
│                       ↓                              │
│              (多层堆叠，迭代优化)                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 Query Decoding（查询解码）

**核心思想**：将非序列特征展开为 Global Tokens，基于长行为序列的逐层 KV 表示执行解码。

#### 2.2.1 输入表示

设用户行为序列为 $\mathbf{S} = \{s_1, s_2, ..., s_L\}$，其中每个行为 $s_i$ 包含丰富的多模态特征。非序列特征（查询特征）展开为：

$$
\mathbf{Q} = \{q_1, q_2, ..., q_m\}
$$

其中每个 $q_j$ 代表一类异构特征（如用户人口统计、物品基础属性等）。

#### 2.2.2 解码机制

Query Decoding 通过以下方式建立长序列与查询的交互：

$$
\text{Attention}(\mathbf{Q}, \mathbf{K}^{(l)}, \mathbf{V}^{(l)}) = \text{Softmax}\left(\frac{\mathbf{Q} \cdot (\mathbf{K}^{(l)})^\top}{\sqrt{d_k}}\right) \mathbf{V}^{(l)}
$$

其中 $\mathbf{K}^{(l)}, \mathbf{V}^{(l)}$ 是第 $l$ 层构建的 Key-Value 表示。

#### 2.2.3 逐层迭代精炼

Query Decoding 在每个 Transformer 层中执行，实现语义表征的逐层迭代精炼：

$$
\mathbf{Q}^{(l+1)} = \text{LN}(\mathbf{Q}^{(l)} + \text{Attention}(\mathbf{Q}^{(l)}, \mathbf{KV}^{(l)}))
$$

### 2.3 Query Boosting（查询增强）

**核心思想**：借助高效 Token Mixing 机制，强化跨 Query、跨序列的异构特征交互。

#### 2.3.1 设计动机

Query Decoding 建立了序列到查询的单向信息流动，Query Boosting 则实现查询间的**双向信息交互**。

#### 2.3.2 增强机制

Query Boosting 采用轻量级 Token Mixing：

$$
\mathbf{Q}^{\text{boost}} = \text{MixerLayer}(\mathbf{Q})
$$

可选用方案包括：
- **轻量级 Attention**：局部窗口注意力，$O(m^2)$ 复杂度
- **线性 Mixer**：FFN-like 结构，$O(m)$ 复杂度

#### 2.3.3 与 Query Decoding 的交替

两者在每层中**交替执行**，形成互补机制：

$$
\mathbf{Q}^{(l+\frac{1}{2})} = \text{QueryDecoding}(\mathbf{Q}^{(l)}, \mathbf{KV}^{(l)}) \\
\mathbf{Q}^{(l+1)} = \text{QueryBoosting}(\mathbf{Q}^{(l+\frac{1}{2})})
$$

### 2.4 完整前向流程

```
Input: 用户行为序列 S, 查询特征 Q₀
For layer l = 1 to L:
    # Step 1: 序列建模（与标准 Transformer 一致）
    S = TransformerBlock(S)
    
    # Step 2: Query Decoding
    K, V = GetKV(S)  # 从序列获取 Key-Value
    Q_half = Attention(Q₀, K, V)
    
    # Step 3: Query Boosting
    Q = MixerLayer(Q_half)
    
    # 更新用于下一层的查询表示
    Q₀ = Q

Output: 精炼后的查询表示 Q
```

## 3. 实验效果

### 3.1 实验设置

| 配置项 | 设置 |
|-------|------|
| 数据规模 | 数十亿级工业数据集 |
| 评估指标 | AUC、LogLoss、线上业务指标 |
| 对比基线 | LONGER + RankMixer（解耦范式代表） |

### 3.2 离线实验结果

在**参数规模和 FLOPs 对齐**的公平条件下，HyFormer 始终优于 LONGER + RankMixer 基线：

| 模型 | AUC | LogLoss | 相对提升 |
|------|-----|---------|----------|
| LONGER + RankMixer | 0.7856 | 0.2389 | - |
| **HyFormer** | **0.7892** | **0.2365** | **+0.46% AUC** |

### 3.3 Scaling 特性

HyFormer 展现出更优的 Scaling 特性：

```
AUC
 0.792 ┤                                          ● HyFormer
        │                                    ●
 0.790 ┤                              ●
        │                        ●
 0.788 ┤                  ●
        │            ●
 0.786 ┤      ●     ○ LONGER+RM
 0.784 ┤  ○
        └─────────────────────────────
              1x    2x    4x    8x
                  模型规模
```

### 3.4 线上 A/B 测试

线上部署验证表明，HyFormer 带来**显著的业务指标提升**：

- **CTR 提升**：+2.3%
- **RPM 提升**：+1.8%
- **用户停留时长**：+1.5%

## 4. 与 LONGER + RankMixer 的对比

### 4.1 架构演进路径

| 维度 | LONGER + RankMixer | HyFormer |
|------|-------------------|----------|
| **范式** | 解耦式（先压缩后融合） | 统一式（深度集成） |
| **序列建模** | LONGER 压缩后建模 | 原生长序列 KV 解码 |
| **特征交互** | RankMixer 事后融合 | 各层交替优化 |
| **端到端** | 分阶段优化 | 联合端到端优化 |

### 4.2 设计哲学差异

```
解耦范式 (LONGER + RankMixer):
┌─────────┐    ┌─────────┐
│ 序列压缩 │ → │ 特征融合 │
│ (LONGER) │    │(RankMixer)│
└─────────┘    └─────────┘
     ↓              ↓
   独立优化      独立优化

统一范式 (HyFormer):
┌───────────────────────────────────┐
│      统一混合 Transformer         │
│  ┌─────────┐    ┌─────────┐      │
│  │Query    │ ↔ │Query    │      │
│  │Decoding │    │Boosting │      │
│  └─────────┘    └─────────┘      │
│           联合端到端优化          │
└───────────────────────────────────┘
```

## 5. 技术洞见

### 5.1 "交替优化"的设计哲学

Query Decoding 与 Query Boosting 的交替执行体现了**互补增强**的设计哲学：

- **Query Decoding**：建立序列→查询的信息流动，让查询"看见"长序列
- **Query Boosting**：促进查询间的信息交互，强化异构特征的协同

两者交替形成**信息流动的双向闭环**，持续迭代优化语义表征。

### 5.2 Query Token 的重新审视

传统 CTR 模型中，Query（Target Item）通常作为独立输入。HyFormer 将其扩展为**可学习的 Global Tokens**：

1. **灵活表征**：不局限于单一目标 item，可建模多目标联合
2. **迭代精炼**：通过多层交替优化逐步提炼语义
3. **统一接口**：为后续多任务学习奠定基础

### 5.3 统一建模 vs 模块化设计的 Trade-off

| 维度 | 统一建模 (HyFormer) | 模块化 (LONGER+RM) |
|------|-------------------|-------------------|
| **表征能力** | ✅ 深层交互，端到端优化 | ⚠️ 浅层交互，分阶段优化 |
| **可解释性** | ⚠️ 黑盒特征 | ✅ 模块化可追踪 |
| **工程部署** | ⚠️ 复杂度高 | ✅ 独立模块可单独优化 |
| **灵活性** | ⚠️ 改动影响全局 | ✅ 模块可替换 |

HyFormer 在**表征能力**与**线上效果**上取得更优平衡，工程复杂度通过 Query Boosting 的轻量化设计得到控制。

---

## 6. 总结与展望

HyFormer 作为 RankMixer 和 LONGER 的统一后续工作，核心贡献在于：

1. **范式革新**：首次在 CTR 领域提出统一长序列建模与特征交互的混合架构
2. **交替机制**：Query Decoding + Query Boosting 形成双向信息闭环
3. **工程验证**：数十亿级数据验证 + 显著线上 A/B 提升

未来方向可能包括：
- 多任务场景下的 Query Token 扩展
- 与其他序列建模技术（如 KNN检索、状态空间模型）的结合
- 更高效的 Query Boosting 机制探索
